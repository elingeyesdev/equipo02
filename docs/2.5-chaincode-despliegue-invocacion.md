# 2.5 Chaincode — despliegue e invocación

Este documento cierra el entregable **2.5**: separación **integrador público** vs **administración**, modelo de solicitud único, invocación vía middleware con lista blanca, respuestas uniformes, flujo de despliegue en test-network, scripts oficiales, bitácora y pruebas.

## 1. Matriz pública vs administración

| Ámbito | Quién | Rutas HTTP | Contrato OpenAPI | Autenticación |
|--------|--------|------------|------------------|---------------|
| **Integrador** | Aplicaciones de terceros documentadas | `POST /chaincode/invocar` | Sí (`openapi.yaml`) | Ninguna en esta versión (solo política de lista blanca). |
| **Administración** | Operadores del equipo / automatización interna | `POST /admin/chaincode/invocar` | No (excluidas explícitamente del validador OpenAPI) | Cabecera `X-Admin-Api-Key` igual a `ADMIN_API_KEY` en el servidor. |

### Funciones permitidas (por contrato + función)

Las reglas efectivas están en `api-middleware/internal/chaincodepolicy/politicas_chaincode.json` (o en archivo externo referenciado por `CHAINCODE_POLITICAS_FILE`). Resumen:

- **Integrador**: principalmente **lecturas** (`evaluate`) sobre `cliente_cc` (p. ej. `ReadAsset`, `GetAllAssets`, `AssetExists`) y consultas del token ERC-20 (`BalanceOf`, `Name`, `Symbol`, `Allowance`, etc.). **No** se expone invoke genérico: toda tupla `(canal, contrato, función, modo, número de argumentos)` debe coincidir con una fila de la lista blanca.
- **Administración**: todo lo del integrador **más** escrituras (`submit`) como `CreateAsset`, `Mint`, `Transfer`, `InitLedger`, etc. En el JSON, la clave `administracion` contiene **solo reglas adicionales**; al cargar, el middleware concatena `integrador` + `administracion` para obtener la lista blanca efectiva de `POST /admin/chaincode/invocar`.

Operaciones de **lifecycle** (package, install, approve, commit) y uso de la **CLI** de Fabric **no** pasan por el middleware: son la “ruta administrativa” descrita en la sección 4 y en `scripts/fabric-despliegue/`.

## 2. Formato de solicitud (JSON único)

```json
{
  "canal": "clientes",
  "contrato": "cliente_cc",
  "funcion": "ReadAsset",
  "parametros": ["CLI001"],
  "modo": "evaluate"
}
```

- **modo**: `evaluate` (respuesta tipo `RespuestaLectura`) o `submit` (respuesta tipo `RespuestaExitoTx` con `txId`).
- **parametros**: arreglo de cadenas; el número debe coincidir **exactamente** con la política para esa función.

### Códigos HTTP habituales

| Código | Uso |
|--------|-----|
| 400 | JSON inválido, `modo` incorrecto. |
| 403 | Combinación no permitida por lista blanca (`POLITICA_NO_PERMITIDA`, p. ej. función/argumentos/modo no listados) o API key administrativa incorrecta. |
| 404 | Claves o entidades inexistentes en ledger (heurística sobre mensajes Fabric). |
| 502 | Endoso / commit rechazado (`FABRIC_TRANSACCION`). |
| 503 | Gateway no inicializado, timeout o red no disponible. |
| 500 | Otros errores Fabric mapeados a mensaje genérico (sin rutas locales ni credenciales en la respuesta JSON). |

## 3. Bitácora (auditoría)

Cada invocación genera una línea JSON en **stdout** con prefijo `[BITACORA_CHAINCODE]` y, si está definido `CHAINCODE_BITACORA_FILE`, la misma línea en **append** al archivo (p. ej. `api-middleware/logs/bitacora-chaincode.log`).

Campos fijos: `timestamp`, `actor` (cabecera opcional `X-Actor-Id`), `audiencia` (`integrador` | `administracion`), `canal`, `contrato`, `funcion`, `modo`, `resultado`, `codigoHttp`, `codigoNegocio`, `txId`, `mensaje` (detalle técnico del error en servidor).

El directorio `api-middleware/logs/` está ignorado por git salvo `.gitignore`.

## 4. Flujo de despliegue en test-network (autorización y artefactos)

1. **Autorización**: los comandos `peer lifecycle` se ejecutan con identidades administrativas de cada organización (p. ej. `Admin@org1.example.com`, `Admin@org2.example.com` en el sample). Cada org **aprueba** la definición del chaincode para su peer; el **commit** en el canal requiere haber alcanzado el umbral de la política de endoso del canal (en el sample suele ser mayoría de orgs del canal).
2. **Artefactos**: paquete `.tar.gz` generado por el empaquetado, **sequence** incremental por nombre de chaincode en el canal, **endorsement policy** opcional (`-ccep` en `network.sh` si aplica).
3. **Secuencia típica** (resumida): levantar red y canal (`./network.sh up createChannel ...`), luego `deployCC` con `-ccn`, `-ccp`, `-ccv`, `-ccs`, `-c` (canal). Ver `red-hyperledger/test-network/scripts/deployCC.sh` y `network.sh deployCC`.

No se asume Kubernetes: el flujo por defecto es **test-network**.

## 6. Ejemplos `curl` (integrador)

**Evaluate feliz** (requiere red, gateway y asset `CLI001`):

```bash
curl -sS -X POST "http://localhost:3000/chaincode/invocar" \
  -H "Content-Type: application/json" \
  -H "X-Actor-Id: demo-integrador" \
  -d '{"canal":"clientes","contrato":"cliente_cc","funcion":"ReadAsset","parametros":["CLI001"],"modo":"evaluate"}'
```

**403 — función no permitida para integrador** (p. ej. `Mint` solo en política admin):

```bash
curl -sS -X POST "http://localhost:3000/chaincode/invocar" \
  -H "Content-Type: application/json" \
  -d '{"canal":"clientes","contrato":"token_erc20","funcion":"Mint","parametros":["10"],"modo":"submit"}'
```

**400 — número de parámetros incorrecto**:

```bash
curl -sS -X POST "http://localhost:3000/chaincode/invocar" \
  -H "Content-Type: application/json" \
  -d '{"canal":"clientes","contrato":"cliente_cc","funcion":"ReadAsset","parametros":[],"modo":"evaluate"}'
```

## 7. Ejemplo administración (`submit`)

Sustituya la clave por la misma que `ADMIN_API_KEY` en el `.env` del servidor:

```bash
curl -sS -X POST "http://localhost:3000/admin/chaincode/invocar" \
  -H "Content-Type: application/json" \
  -H "X-Admin-Api-Key: cambiar-en-produccion" \
  -H "X-Actor-Id: operador-local" \
  -d '{"canal":"clientes","contrato":"token_erc20","funcion":"Mint","parametros":["1"],"modo":"submit"}'
```

## 8. Pruebas automatizadas

Desde `api-middleware/`:

```bash
go test ./internal/handlers/... -count=1
```

Incluyen casos de validación, política y gateway no inicializado sin necesidad de red levantada.

Para validación integral con Fabric, use los `curl` anteriores con la red en marcha y el middleware conectado (`fabric.Connect` exitoso en arranque).
